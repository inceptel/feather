# Shared chats across users and boxes ("Peers")

Status: phases 1–3 implemented 2026-06-11 (inbound /api/share, peer-box
share client, frontend box/peer selector + share menu). Phase 4 (cross-box
aggregation) and phase 5 (hardening) outstanding. Companion audit findings
at the bottom.

## Goal

Let two Feather users share session transcripts with each other, with
per-peer granularity:

- **allan ↔ phil**: share everything, both directions, and **talk
  together** — either of us can watch a session live *and* send messages
  into it (a group chat with the agent).
- **allan ↔ zak**: share only explicitly selected sessions, view-only.

Sharing must span *all of a user's boxes* (testbox, mac-mini, scout,
personal, …), not just the switchboard's local sessions, and must be safe:
a peer must never gain write/control/filesystem access, and a peer with a
"selected" policy must never see anything beyond their grants.

## Core idea: a peer is a box with a passport

Feather already has exactly the right abstraction: `boxes.json` maps an id
to a remote Feather URL, and the `?box=` middleware proxies session routes
to it (`server.js:845`). Sharing reuses this in both directions:

- **Outbound (viewing someone else's chats):** a peer is just another entry
  in `boxes.json`, marked `"peer": true`, with a bearer token. It shows up
  in the box list like any box; selecting it shows the sessions that user
  shared with you. Read-only.
- **Inbound (someone viewing yours):** a new, deliberately tiny API surface
  `/api/share/*` that requires a peer token, applies your sharing policy,
  and aggregates across your own boxes before answering.

No in-app human auth is added. Humans still authenticate via
Authelia/Traefik; the peer token authenticates a *server*, not a person.
Each instance has a single owner, declared in config.

## Config

### `boxes.json` (already gitignored) — outbound entries gain two fields

```json
{
  "scout":    { "url": "http://10.88.0.15:9000", "label": "Scout" },
  "personal": { "url": "http://10.88.0.26:9000", "label": "Personal" },
  "phil":     { "url": "http://10.88.0.31:4870", "label": "Phil",
                "peer": true, "token": "<random-32-bytes-hex>" }
}
```

A `peer: true` box is routed through the **share client**, not the general
proxy: only `GET` to `/api/share/sessions`, `…/messages`, `…/stream`,
`…/export` is ever sent, with `Authorization: Bearer <token>`. The general
proxy (`proxyToBox`) refuses peer boxes, so `send`/`resume`/`delete`/
`interrupt`/terminal can never be forwarded to a peer even by URL-crafting.

### `sharing.json` (new, gitignored, mode 0600) — inbound policy

```json
{
  "owner": "allan",
  "peers": {
    "phil": { "token": "<hex>", "policy": "all", "control": true },
    "zak":  { "token": "<hex>", "policy": "selected" }
  },
  "grants": [
    { "peer": "zak", "box": "local", "session": "8f2e41ac-…" },
    { "peer": "zak", "box": "scout", "session": "0b7cd2e9-…" },
    { "peer": "zak", "box": "*",     "project": "-home-user-hoa" }
  ]
}
```

- `policy: "all"` — peer sees every non-worker session on every box this
  switchboard can reach (same set the owner sees in their own list).
- `control: true` — peer can also **send into** shared sessions (and
  interrupt them). This is the "talk together" mode: both humans watch the
  same SSE stream and both can type. Reserved for fully-trusted peers;
  never combine with `policy: "selected"` semantics loosely — a control
  peer can instruct an agent running with `--dangerously-skip-permissions`,
  which is equivalent to shell access on that box.
- `policy: "selected"` — peer sees only sessions matched by a grant.
  Grants match by `(box, session)` or `(box, project)`; `"*"` box means all
  boxes. Default is **deny**: no grant, no access.
- Tokens are long random secrets generated by a small CLI helper
  (`node server.js --add-peer zak`), compared with
  `crypto.timingSafeEqual`. Never committed (file is gitignored; add it to
  `.gitignore` alongside `boxes.json`).

Per-session quick grants also work from the UI: a "Share…" menu on a
session writes `share: ["zak"]` into `session-meta.json`; the inbound
filter treats that as an implicit grant for local sessions. `sharing.json`
grants cover remote-box sessions and project-wide rules.

## Inbound API surface (the only thing a peer can call)

All under `/api/share`, behind `requirePeer` middleware:

```
GET  /api/share/sessions                 → filtered, aggregated session list
GET  /api/share/sessions/:id/messages    → transcript (existing parser)
GET  /api/share/sessions/:id/stream      → SSE live tail
GET  /api/share/sessions/:id/export      → markdown export

# control: true peers only (403 otherwise):
POST /api/share/sessions/:id/send        → routes to sendInput()
POST /api/share/sessions/:id/interrupt   → C-c the tmux session
```

**Attribution:** the Claude transcript has no concept of *which human*
typed a message, so peer sends are prefixed in the text itself:
`sendInput(id, "[phil] " + text)`. That makes the speaker visible to the
agent (it can address each of us) and to both UIs in the rendered
transcript, with zero schema changes. The owner's own sends stay
unprefixed. The share-access log records every peer send verbatim.

`requirePeer`:
1. Reads `Authorization: Bearer <token>`, resolves to a peer id via
   constant-time compare against `sharing.json`. 401 otherwise.
2. Attaches `req.peer = { id, policy }`.
3. Logs `{ts, peer, route, session}` to `~/.feather/share-access.log`.

`/api/share/sessions` aggregation: run local `discoverSessions()`, plus
fan out to own non-peer boxes (`GET {box.url}/api/sessions`, the existing
endpoint), tag each result with `box`, then filter through the grant check.
Session ids returned to peers are namespaced `box:sessionId` so the
follow-up message/stream calls know where to route — and the grant check
runs again on every call (list filtering alone is not enforcement).

For `messages`/`stream`/`export` of a remote-box session, the switchboard
re-proxies to the box using the existing internal endpoints *after* the
grant check passes. Boxes themselves need zero changes — all policy lives
on the owner's switchboard, one file to manage, one place to revoke.

Explicitly **excluded** from the share surface, permanently (even for
control peers): `resume`, `delete`, `fork`, `rename`, `/api/file`,
`/api/files`, `/api/terminal`, `/api/shell`, `/api/upload`, `/api/auto`,
`/api/cos`. Send/interrupt exist only in the gated form above. A shared
transcript's linkified file paths must not resolve for a peer (the viewer
routes file links through the *viewer's own* `/api/file`, which will
404 — acceptable; do not proxy file reads).

## Outbound UI

- Box selector (sidebar) lists own boxes, then peers under a "Shared with
  me" divider with a distinct icon.
- Selecting a peer calls `/api/sessions?box=phil`, which the middleware
  routes through the share client → `GET {phil.url}/api/share/sessions`.
- Session view in peer mode hides the terminal tab and resume/fork/delete
  buttons. The composer is shown iff the peer granted us `control` (the
  share list response includes a `control: true/false` capability flag);
  sends go to `POST /api/share/sessions/:id/send`. With control, a shared
  session is a three-way conversation: both humans plus the agent, live.
- On the owner's own sessions, a share badge shows who can see it.

## Transport between instances

Peers on the same `10.88.0.x` podman bridge talk directly (current model).
Cross-host peers go through `https://<user>.feather-cloud.dev` with a
Traefik/Authelia **bypass rule scoped to `/api/share`** — safe because that
path is token-checked in-app and read-only by construction. Nothing else
bypasses Authelia.

## Why this shape

- **Integrates with boxes.json** rather than inventing a parallel registry;
  the frontend box selector work pays off for both own-boxes and peers.
- **Enforcement at the data owner**, not the viewer: Zak's client never
  receives anything his token doesn't grant, so a malicious/buggy viewer
  can't widen access.
- **Single policy point** (switchboard) even though sessions live on many
  boxes — matches "share across many boxes" without touching every box.
- **Capability-gated control**: viewing and talking are separate grants.
  Zak-tier peers are view-only by construction; allan↔phil get `control`
  and can co-drive sessions. The dangerous verbs (filesystem, terminal,
  lifecycle) stay owner-only regardless.

## Safety prerequisites found during the audit

These should land with (or before) sharing, since sharing widens exposure:

1. **Bridge exposure**: `server.js` listens on `0.0.0.0:4870`
   (`server.js:1928`) with zero auth — any container on the bridge already
   has full control (including `DELETE /api/file` recursive delete at
   `server.js:1114` and the `/api/shell` PTY at `server.js:1897`). Once
   peers exist on the bridge, the *full* API must not be reachable by them:
   either bind to localhost + Traefik only, or require a local-secret
   header for everything outside `/api/share`.
2. **`/api/file` / `/api/files` / `DELETE /api/file`** accept any absolute
   path (only check `startsWith('/')`). Add an allowlist root set (HOME,
   uploads) — independent of sharing, but mandatory before any semi-trusted
   party can reach the port.
3. **SSRF via boxes.json** (`proxyToBox`, `server.js:47`): URL is used
   verbatim. Low risk while the file is owner-edited; the share client
   restricts methods/paths for peer entries anyway.
4. **Transcripts can contain secrets** (env vars echoed in tool output,
   tokens in command lines). UI should show a one-time confirm when
   granting a session to a "selected"-policy peer, and `policy: "all"`
   should be reserved for fully-trusted peers.

## Implementation order

1. `sharing.json` + `requirePeer` + `/api/share/sessions[…]` (local
   sessions only) + gated send/interrupt with `[name]` attribution +
   access log. ~180 lines in server.js, no schema changes.
2. Share client for `peer: true` boxes + middleware guard in
   `proxyToBox`. ~40 lines.
3. Frontend: box/peer selector, read-only session view, share menu writing
   `session-meta.json.share`. (The box selector itself is currently
   missing from the UI — server-side `?box=` support exists but nothing in
   `frontend/src/api.ts` sends it.)
4. Cross-box aggregation in `/api/share/sessions` + `box:id` namespacing.
5. Hardening items 1–2 above.
